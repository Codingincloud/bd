\chapter{SYSTEM ANALYSIS}

\section{Introduction}

System analysis is a critical phase in software development that involves studying the current system, identifying requirements, and determining the feasibility of the proposed solution. This chapter presents a comprehensive analysis of the Blood Donor Information Management System (BDIMS), including feasibility study, requirement analysis, and system specifications.

\section{Feasibility Study}

A feasibility study determines whether the proposed system is viable from technical, economic, operational, and schedule perspectives.

\subsection{Technical Feasibility}

Technical feasibility examines whether the project can be implemented with available technology and technical resources.

\subsubsection{Hardware Requirements}

\textbf{Development Environment:}
\begin{itemize}
    \item Processor: Intel Core i5 or equivalent
    \item RAM: Minimum 8 GB
    \item Storage: 256 GB SSD or higher
    \item Internet Connection: Stable broadband (minimum 2 Mbps)
\end{itemize}

\textbf{Deployment Server:}
\begin{itemize}
    \item Processor: Dual-core CPU (2.0 GHz or higher)
    \item RAM: Minimum 4 GB (8 GB recommended)
    \item Storage: 50 GB SSD with backup
    \item Network: High-speed internet with static IP
\end{itemize}

\textbf{Client Requirements:}
\begin{itemize}
    \item Any device with a modern web browser
    \item Internet connection (minimum 1 Mbps)
    \item Screen resolution: Minimum 375x667 (mobile), recommended 1920x1080 (desktop)
\end{itemize}

\subsubsection{Software Requirements}

\textbf{Development Tools:}
\begin{itemize}
    \item Operating System: Windows 10/11, macOS 10.15+, or Linux (Ubuntu 20.04+)
    \item Programming Language: Python 3.12+
    \item Framework: Django 5.2.8
    \item Database: PostgreSQL 13+
    \item Code Editor: VS Code, PyCharm, or similar IDE
    \item Version Control: Git 2.30+
    \item Browser: Chrome, Firefox, Safari, or Edge (latest versions)
\end{itemize}

\textbf{Python Libraries:}
\begin{itemize}
    \item Django: Web framework
    \item psycopg2: PostgreSQL adapter
    \item Pillow: Image processing
    \item python-decouple: Configuration management
    \item whitenoise: Static file serving
\end{itemize}

\textbf{Frontend Technologies:}
\begin{itemize}
    \item HTML5: Structure and semantics
    \item CSS3: Styling and responsive design
    \item JavaScript (ES6+): Client-side interactivity
    \item Leaflet.js: Interactive mapping
    \item Chart.js: Data visualization
    \item Font Awesome: Icon library
\end{itemize}

\subsubsection{Technical Expertise}

The development team possesses:
\begin{itemize}
    \item Proficiency in Python programming
    \item Understanding of Django framework and MVT architecture
    \item Experience with PostgreSQL database management
    \item Knowledge of HTML, CSS, and JavaScript
    \item Familiarity with Git version control
    \item Understanding of web security principles
\end{itemize}

\textbf{Conclusion:} The project is \textbf{technically feasible} with available technology, tools, and team expertise.

\subsubsection{Cost-Benefit Analysis}

\textbf{Tangible Benefits:}
\begin{itemize}
    \item Reduced staff time: 10-15 hours per week saved on manual record keeping
    \item Reduced paper costs: NPR 50,000 annually
    \item Improved donor retention: 20-30\% increase in repeat donations
    \item Faster emergency response: Reduction from hours to minutes
\end{itemize}

\textbf{Intangible Benefits:}
\begin{itemize}
    \item Improved donor satisfaction and trust
    \item Enhanced organizational reputation
    \item Better decision-making through data analytics
    \item Increased blood availability for emergencies
    \item Lives saved through efficient blood management
\end{itemize}

\textbf{Conclusion:} The project is \textbf{economically feasible} with low implementation costs and significant long-term benefits.

\subsection{Operational Feasibility}

Operational feasibility examines whether the system will be accepted and used effectively by stakeholders.

\subsubsection{User Acceptance}

\begin{itemize}
    \item \textbf{Administrators:} Healthcare staff are increasingly comfortable with digital systems and appreciate tools that reduce manual work.
    \item \textbf{Donors:} Modern donors, especially younger demographics, prefer digital interactions over phone calls and paper forms.
    \item \textbf{Management:} Hospital management supports digitization initiatives that improve efficiency and patient care.
\end{itemize}

\subsubsection{Training Requirements}

\begin{itemize}
    \item \textbf{Administrators:} 2-3 hours of training covering system navigation, donor management, and report generation
    \item \textbf{Donors:} Minimal training required due to intuitive interface; online help documentation available
    \item \textbf{IT Staff:} 4-6 hours of technical training for deployment, maintenance, and troubleshooting
\end{itemize}

\subsubsection{Change Management}

\begin{itemize}
    \item Gradual migration from existing system with parallel operation period
    \item Data import from existing records (Excel/paper) into BDIMS
    \item User feedback collection during pilot phase
    \item Continuous support during transition period
\end{itemize}

\textbf{Conclusion:} The project is \textbf{operationally feasible} with high user acceptance and manageable training requirements.

\section{Requirement Analysis}

Requirement analysis identifies what the system must do (functional requirements) and how well it must perform (non-functional requirements).

\subsection{Functional Requirements}

Functional requirements define specific behaviors and functions of the system.

\subsubsection{User Authentication and Authorization}

\begin{itemize}
    \item \textbf{FR-1.1:} System shall allow new donors to register with email, phone, and password
    \item \textbf{FR-1.2:} System shall authenticate users with username/email and password
    \item \textbf{FR-1.3:} System shall provide password reset functionality via email
    \item \textbf{FR-1.4:} System shall implement role-based access (Admin, Donor)
    \item \textbf{FR-1.5:} System shall log user login/logout activities
    \item \textbf{FR-1.6:} System shall enforce password complexity requirements
\end{itemize}

\subsubsection{Donor Management}

\begin{itemize}
    \item \textbf{FR-2.1:} System shall store comprehensive donor profiles including personal info, blood type, contact details, and location
    \item \textbf{FR-2.2:} System shall allow donors to update their profile information
    \item \textbf{FR-2.3:} System shall allow profile picture uploads
    \item \textbf{FR-2.4:} System shall enable donors to update location via interactive map
    \item \textbf{FR-2.5:} System shall display donor eligibility status on dashboard
    \item \textbf{FR-2.6:} System shall list all donors with search and filter capabilities for administrators
\end{itemize}

\subsubsection{Health Metrics Tracking}

\begin{itemize}
    \item \textbf{FR-3.1:} System shall allow recording of hemoglobin levels, blood pressure, weight, temperature, and pulse rate
    \item \textbf{FR-3.2:} System shall display health metrics history with date-time stamps
    \item \textbf{FR-3.3:} System shall validate health metric entries against acceptable ranges
    \item \textbf{FR-3.4:} System shall calculate eligibility based on health metrics
    \item \textbf{FR-3.5:} System shall provide health measurement guidance
\end{itemize}

\subsubsection{Donation History Management}

\begin{itemize}
    \item \textbf{FR-4.1:} System shall record all donations with date, center, units donated, and health parameters
    \item \textbf{FR-4.2:} System shall display complete donation history to donors
    \item \textbf{FR-4.3:} System shall calculate total donations and last donation date
    \item \textbf{FR-4.4:} System shall determine next eligible donation date (56 days rule)
    \item \textbf{FR-4.5:} System shall allow administrators to add donation records
\end{itemize}

\subsubsection{Donation Request Management}

\begin{itemize}
    \item \textbf{FR-5.1:} System shall allow donors to schedule donation appointments
    \item \textbf{FR-5.2:} System shall check eligibility before accepting requests
    \item \textbf{FR-5.3:} System shall allow administrators to approve/reject donation requests
    \item \textbf{FR-5.4:} System shall track request status (pending, approved, completed, cancelled, rejected)
    \item \textbf{FR-5.5:} System shall notify donors of request status changes
    \item \textbf{FR-5.6:} System shall automatically update inventory upon donation completion
\end{itemize}

\subsubsection{Blood Inventory Management}

\begin{itemize}
    \item \textbf{FR-6.1:} System shall track blood inventory for all 8 blood types
    \item \textbf{FR-6.2:} System shall display available and reserved units separately
    \item \textbf{FR-6.3:} System shall allow administrators to update inventory levels
    \item \textbf{FR-6.4:} System shall alert when inventory falls below threshold (e.g., 10 units)
    \item \textbf{FR-6.5:} System shall maintain inventory update history
    \item \textbf{FR-6.6:} System shall visualize inventory levels with charts
\end{itemize}

\subsubsection{Emergency Request System}

\begin{itemize}
    \item \textbf{FR-7.1:} System shall allow administrators to create emergency blood requests
    \item \textbf{FR-7.2:} System shall match requests with eligible donors by blood type
    \item \textbf{FR-7.3:} System shall filter donors by location proximity
    \item \textbf{FR-7.4:} System shall display emergency requests prominently to matching donors
    \item \textbf{FR-7.5:} System shall track emergency request status
    \item \textbf{FR-7.6:} System shall log donor responses to emergencies
\end{itemize}

\subsubsection{Hospital Management}

\begin{itemize}
    \item \textbf{FR-8.1:} System shall maintain database of hospitals/blood centers
    \item \textbf{FR-8.2:} System shall store hospital details including name, location, contact info
    \item \textbf{FR-8.3:} System shall associate inventory with specific hospitals
    \item \textbf{FR-8.4:} System shall allow administrators to add/update hospital information
\end{itemize}

\subsubsection{Reporting and Analytics}

\begin{itemize}
    \item \textbf{FR-9.1:} System shall generate donor statistics reports
    \item \textbf{FR-9.2:} System shall generate donation history reports
    \item \textbf{FR-9.3:} System shall generate inventory status reports
    \item \textbf{FR-9.4:} System shall display dashboard with key metrics and charts
    \item \textbf{FR-9.5:} System shall allow date range filtering for reports
\end{itemize}

\subsection{Non-Functional Requirements}

Non-functional requirements define system qualities and constraints.

\subsubsection{Performance Requirements}

\begin{itemize}
    \item \textbf{NFR-1.1:} System shall load pages within 3 seconds on standard broadband
    \item \textbf{NFR-1.2:} System shall support at least 100 concurrent users
    \item \textbf{NFR-1.3:} System shall handle database queries efficiently (response time < 500ms)
    \item \textbf{NFR-1.4:} System shall process search queries within 1 second
\end{itemize}

\subsubsection{Security Requirements}

\begin{itemize}
    \item \textbf{NFR-2.1:} System shall store passwords using secure hashing (PBKDF2+SHA256)
    \item \textbf{NFR-2.2:} System shall protect against CSRF attacks
    \item \textbf{NFR-2.3:} System shall prevent SQL injection through parameterized queries
    \item \textbf{NFR-2.4:} System shall implement XSS protection through template escaping
    \item \textbf{NFR-2.5:} System shall use HTTPS for all communications in production
    \item \textbf{NFR-2.6:} System shall enforce role-based access control
    \item \textbf{NFR-2.7:} System shall implement secure session management
\end{itemize}

\subsubsection{Reliability Requirements}

\begin{itemize}
    \item \textbf{NFR-3.1:} System shall have 99\% uptime
    \item \textbf{NFR-3.2:} System shall perform daily database backups
    \item \textbf{NFR-3.3:} System shall handle errors gracefully with user-friendly messages
    \item \textbf{NFR-3.4:} System shall maintain data integrity through transactions
    \item \textbf{NFR-3.5:} System shall log critical errors for debugging
\end{itemize}

\subsubsection{Usability Requirements}

\begin{itemize}
    \item \textbf{NFR-4.1:} System shall have intuitive navigation requiring minimal training
    \item \textbf{NFR-4.2:} System shall provide clear error messages and guidance
    \item \textbf{NFR-4.3:} System shall maintain consistent UI design across all pages
    \item \textbf{NFR-4.4:} System shall be accessible to users with basic computer literacy
    \item \textbf{NFR-4.5:} System shall support keyboard navigation
\end{itemize}

\subsubsection{Compatibility Requirements}

\begin{itemize}
    \item \textbf{NFR-5.1:} System shall work on Chrome, Firefox, Safari, and Edge browsers
    \item \textbf{NFR-5.2:} System shall be responsive on devices from 375px to 2560px width
    \item \textbf{NFR-5.3:} System shall support both desktop and mobile interfaces
    \item \textbf{NFR-5.4:} System shall work on Windows, macOS, Linux, iOS, and Android
\end{itemize}

\subsubsection{Maintainability Requirements}

\begin{itemize}
    \item \textbf{NFR-6.1:} System shall follow PEP 8 coding standards for Python
    \item \textbf{NFR-6.2:} System shall include inline code documentation
    \item \textbf{NFR-6.3:} System shall have modular architecture for easy updates
    \item \textbf{NFR-6.4:} System shall use version control (Git)
    \item \textbf{NFR-6.5:} System shall include user and technical documentation
\end{itemize}

\subsubsection{Scalability Requirements}

\begin{itemize}
    \item \textbf{NFR-7.1:} System shall support up to 10,000 donor records initially
    \item \textbf{NFR-7.2:} System architecture shall allow horizontal scaling
    \item \textbf{NFR-7.3:} Database shall support growth to 100,000+ records
    \item \textbf{NFR-7.4:} System shall allow future feature additions without major refactoring
\end{itemize}

\section{Summary}

This chapter presented a comprehensive system analysis for BDIMS, including feasibility studies that confirmed the project's technical, economic, operational, and schedule viability. Detailed functional requirements specify 50+ specific system behaviors across nine major functional areas, while non-functional requirements define critical system qualities including performance, security, reliability, usability, compatibility, maintainability, and scalability. This thorough analysis provides a solid foundation for the system design phase presented in the next chapter.
